üìä Overview

The Chicago Hub pricing system uses a standardized calculation framework that normalizes all pricing models to a common base (daily rates) and then projects revenue across any timeframe.

Key Principles

Core Concepts:
  • Standardization: All pricing uses flatRate + pricingModel schema
  • Daily Base Unit: All calculations normalize to daily rates first
  • Timeframe Projection: Revenue is projected using days √ó daily_rate
  • Conservative Forecasting: Revenue forecasts use the 1x tier (full price)
  • Volume Discounts: Multi-tier pricing represents volume discounts

Schema Structure

{
  "flatRate": 300,           // The price amount
  "pricingModel": "per_send", // How the price is applied
  "frequency": "4x"           // Commitment multiplier (optional)
}

Pricing vs Revenue

Concept What It Means Example
Pricing What you charge the customer (commitment package total) $300/send √ó 4x = $1,200
Revenue What you'll actually earn over time (forecasted income) $300/send √ó 4.33 sends/month = $1,299/month

üéØ Core Formulas

1. Commitment Package Total
Total = Base Price √ó Frequency Multiplier
Example: $300 √ó 4x = $1,200
Use: Package pricing, customer invoices
2. Revenue Forecast
Revenue = Daily Rate √ó Days in Timeframe
Example: $300 √ó 4.33 occurrences/month = $1,299/month
Use: Financial projections, reporting
3. Hub Discount
Discount % = ((Default - Hub) / Default) √ó 100
Example: (($300-$250)/$300) √ó 100 = 16.67%
Use: Hub pricing calculations
4. Package Bundle Pricing
Package Price = SUM(monthly revenue for each item)
Uses hub pricing when available, otherwise default pricing
Example: $1,083 + $500 + $1,732 = $3,315/month
Use: Multi-publication packages

üè¢ Hub Pricing & Discounts

What is Hub Pricing?

Publications can offer different prices to different marketing hubs. This allows for volume discounts and strategic partnerships.

Discount Calculation

Hub Discount % = ((Default Price - Hub Price) / Default Price) √ó 100
Savings = Default Price - Hub Price

üé• Streaming Video

Streaming video advertising is calculated based on channel-level metrics (views, frequency) combined with ad-specific properties.

üìä Key Metrics

Channel-Level Metrics

  • subscribers: Total channel subscribers
  • averageViews: Average views per video/episode
  • frequency: Publishing frequency (daily, weekly, bi-weekly, monthly)
  • platform: Array of platforms (YouTube, Twitch, Facebook Live, etc.)

Ad-Level Properties

  • position: pre-roll, mid-roll, or post-roll
  • duration: Ad length in seconds
  • spotsPerShow: Number of times ad runs per video (defaults to 1)
  • adFormat: pre-roll, mid-roll, post-roll, overlay, sponsored_content, product_placement, live_mention

üéØ Best Practices

1. Use CPV for Streaming

CPV (per 100 views) is the industry standard for video streaming platforms like YouTube and Twitch. It aligns with typical streaming ad rates.

2. Set spotsPerShow Correctly

If you run multiple ads per video (e.g., 1 pre-roll + 2 mid-rolls), set spotsPerShow to 3. This automatically multiplies your impressions.

3. Multi-Platform Support

Use the platform array to track simultaneous streaming:

"platform": ["youtube", "twitch", "facebook_live"]

4. Channel vs. Ad Metrics

Store metrics at the channel level (averageViews, frequency, subscribers). The system automatically calculates ad-level metrics using:

  • impressionsPerMonth = averageViews √ó frequency √ó spotsPerShow
  • occurrencesPerMonth = frequency √ó spotsPerShow

üîß Troubleshooting Guide

Issue 1: "Total Not Showing"

Problem: Commitment total is hidden

Check:
  • Is pricingModel = "contact"? ‚Üí Total always hidden for contact pricing
  • Is flatRate missing or 0? ‚Üí Add a flatRate value
  • Is frequency empty? ‚Üí This is OK, shows base price (1x)

Issue 2: "Revenue Shows $0"

Problem: Revenue forecast is zero

For Occurrence-Based Models:
  • Add performanceMetrics.occurrencesPerMonth
  • OR provide channelFrequency when calling calculateRevenue()
For Impression-Based Models:
  • Add performanceMetrics.impressionsPerMonth

Issue 3: "Wrong Tier Used"

Expected Behavior: System uses 1x tier for forecasting

Why:
  • Revenue forecasts need conservative estimates
  • 1x tier = base rate (maximum revenue potential)
  • Discounted tiers (4x, 12x) are for customer packages, not forecasting

Issue 4: "Hub Pricing Not Applying"

Problem: Getting default rates instead of hub rates

Checklist:
  • Does hubPricing array exist?
  • Does hub entry with matching hubId exist?
  • Is available = true (or undefined)?
  • Are you passing the hubId parameter?
